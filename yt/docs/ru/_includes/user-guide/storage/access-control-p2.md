## Системные субъекты { #system_subjects }

В {{product-name}} есть встроенные субъекты, необходимые для работы системы. Такие субъекты нельзя удалить:

1. Пользователи: `guest`, `root`, `scheduler` и `job`.
2. Группы: `everyone`, `users` и `superusers`.

## Атрибуты субъектов { #subject_attributes }

Ниже перечислены атрибуты, которые есть у любого субъекта.

| **Атрибут** | **Тип** | **Описание** |
| ------------------- | --------------- | ------------------------------------------------------------ |
| `name` | `string` | Имя субъекта. |
| `member_of` | `array<string>` | Группы, в которые субъект входит напрямую. |
| `member_of_closure` | `array<string>` | Группы, в которые субъект входит напрямую или косвенно. |
| `aliases` | `array<string>` | Альтернативные имена, на которые можно ссылаться в ACL. |

## Атрибуты пользователей { #user_attributes }

Помимо общих атрибутов субъекта, у пользователя есть атрибуты, перечисленные ниже.

| **Атрибут** | **Тип** | **Описание** | **Обязательный** |
| -------------------------- | --------------- | ------------------------------------------------------------ | ------------------- |
| `banned` | `bool` | Признак блокировки пользователя. | Нет |
| `access_time` | `DateTime` | Время последнего запроса от пользователя. | Да |
| `access_counter` | `integer` | Общее число запросов пользователя. | Да |
| `request_rate` | `double` | Текущая частота запросов в секунду. | Да |
| `request_rate_limit` | `double` | Ограничение на число запросов в секунду. По умолчанию 100. | Да |
| `request_queue_size_limit` | `double` | Ограничение на размер очереди запросов. По умолчанию 100. | Да |
| `usable_accounts` | `array<string>` | Аккаунты, которые пользователь может использовать. | Да |

## Атрибуты групп { #group_attributes }

Помимо общих атрибутов субъекта, у групп есть следующие атрибуты.

| **Атрибут** | **Тип** | **Описание** |
| ----------- | --------------- | --------------------------------------------------------- |
| `members` | `array<string>` | Члены группы: пользователи и другие группы. |

## Управление группами { #group_control }

{% note info "Примечание" %}

На больших кластерах прямое управление пользователями и группами обычно доступно только администраторам {{product-name}}.

{% endnote %}

Чтобы создать группу, используйте команду `create` и задайте атрибут `name`:

```bash
yt create group --attributes '{name=my_group}'
```

Чтобы добавить пользователя или другую группу в группу, используйте `add-member`:

```bash
yt add-member my_user my_group
```

Чтобы удалить участника из группы, используйте `remove-member`:

```bash
yt remove-member my_user my_group
```

Чтобы удалить группу, удалите соответствующий узел из `//sys/groups`:

```bash
yt remove //sys/groups/my_group
```

При удалении группы она автоматически удаляется из всех ACL, где на неё есть ссылки.

## Авторизация { #authorization }

Модуль авторизации на мастер-сервере отвечает на вопрос: следует ли выдать пользователю `U` право `P` на объект `O`?

- `U` — пользователь. Группа не может инициировать запрос, но её членство учитывается при проверке.
- `P` — право доступа.
- `O` — системный объект, например узел Кипариса, аккаунт, транзакция, операция, пользователь или группа.

Если `U` равен `root`, запрос всегда разрешается.

Права, поддерживаемые общей моделью ACL, перечислены ниже.

| Право | Описание |
| ------------ | ------------------------------------------------------------ |
| `read` | Чтение данных объекта или его атрибутов. |
| `write` | Изменение состояния объекта или его атрибутов. |
| `use` | Использование аккаунта, пула или бандла. |
| `administer` | Изменение дескриптора управления доступом объекта. |
| `create` | Создание объектов заданного схематического типа. |
| `remove` | Удаление объекта. |
| `mount` | Монтирование, размонтирование, перемонтирование и решардирование динамической таблицы. |
| `manage` | Управление операцией или её джобами. |

Чтобы принять решение, {{product-name}} строит **эффективный ACL** объекта `O`. Эффективный ACL состоит из:

- ACL, заданного непосредственно на объекте;
- унаследованных ACE от родительских объектов, если `@inherit_acl` равно `%true`.

Порядок ACE внутри ACL значения не имеет.

Каждая ACE имеет следующую структуру.

| **Атрибут** | **Тип** | **Описание** |
| ------------------ | ------------------- | ------------------------------------------------------------ |
| `action` | `SecurityAction` | `allow` или `deny`. |
| `subjects` | `array<string>` | Субъекты, на которые распространяется ACE. |
| `permissions` | `array<Permission>` | Права, к которым применяется ACE. |
| `inheritance_mode` | `InheritanceMode` | Режим наследования ACE. По умолчанию `object_and_descendants`. |

Параметр `inheritance_mode` задаёт область действия ACE.

| **Значение** | **Действие** |
| --------- | ---------- |
| `object_only` | ACE действует только на текущий объект. |
| `object_and_descendants` | ACE действует на текущий объект и всех потомков. |
| `descendants_only` | ACE действует только на потомков, включая непрямых. |
| `immediate_descendants_only` | ACE действует только на прямых потомков. |

ACE считается применимой к `U` и `P`, если:

1. `P` присутствует в списке `permissions`.
2. В `subjects` указан либо сам пользователь `U`, либо группа, в которую `U` входит прямо или косвенно.

После построения эффективного ACL результат определяется так:

1. Доступ выдается, если существует хотя бы одна применимая ACE с `allow` и не существует ни одной применимой ACE с `deny`.
2. Во всех остальных случаях доступ запрещается.

Это означает, что пустой эффективный ACL по умолчанию запрещает доступ.

## Типовые операции с ACL { #acl_tasks }

Чтобы посмотреть дескриптор управления доступом, используйте следующие атрибуты:

```bash
yt get //home/project/@acl
yt get //home/project/@inherit_acl
yt get //home/project/@owner
```

Чтобы добавить ACE в конец существующего ACL, используйте специальный элемент `end`:

```bash
yt set //home/project/@acl/end '{action=allow; subjects=[analysts]; permissions=[read]}'
```

Чтобы полностью заменить ACL, задайте весь список целиком:

```bash
yt set //home/project/@acl '[{action=allow; subjects=[analysts]; permissions=[read]}; {action=allow; subjects=[admins]; permissions=[read; write; administer; remove]}]'
```

Чтобы запретить наследование ACL от родителей, установите `@inherit_acl` в `%false`:

```bash
yt set //home/project/@inherit_acl %false
```

Чтобы проверить наличие у пользователя определённого права на узел Кипариса, используйте `check-permission`:

```bash
yt check-permission <user_name> <permission> <path>
```

Пример:

```bash
$ yt check-permission pavel-kulenov write //tmp
{
  "action" = "allow";
  "object_id" = "1-3-411012f-1888ce1f";
  "object_name" = "node //tmp";
  "subject_id" = "c4-8aaa-41101f6-bec6113b";
  "subject_name" = "YTsaurus";
}
```

## Владелец объекта и субъект `owner` { #owner}

Когда пользователь `U` создаёт объект, он становится его владельцем. Имя владельца хранится в атрибуте `@owner`.

{% note info "Примечание" %}

Изменить владельца может только суперпользователь, то есть участник группы `superusers`.

{% endnote %}

В {{product-name}} также есть специальный псевдо-субъект `owner`. Аутентифицироваться под ним нельзя, но на него можно ссылаться в ACL. Во время авторизации `owner` подставляется фактическим владельцем объекта.

Этот субъект удобно использовать для правил вида «удалять объекты в каталоге может только тот, кто их создал»:

```bash
{action=allow; permissions=[remove]; subjects=[owner]; inheritance_mode=descendants_only}
```

В подобных случаях отключайте наследование ACL, если правила родителей не должны влиять на потомков:

```bash
yt set //home/project/@inherit_acl %false
```

## Управление операциями

У операций есть собственные ACL, как и у узлов Кипариса. Посмотреть ACL операции можно в атрибуте `runtime_parameters/acl`.

Основные права для операций имеют следующий смысл:

- `read` даёт доступ к live preview, промежуточным данным, stderr, fail context и входным данным джобов;
- `manage` разрешает действия, меняющие состояние операции или джобов, например `Abort`, `Complete`, `Suspend`, `Resume`, `Abandon` и `Send signal`;
- [Job Shell](../../../user-guide/problems/jobshell-and-slowjobs.md) требует одновременно `read` и `manage`, поскольку доступ к оболочке позволяет и читать данные, и менять состояние.

ACL операции можно задать при её запуске:

```python
yt.run_map(..., spec={"acl": [{
 "action": "allow",
 "subjects": ["u1", "u2"],
 "permissions": ["read", "manage"],
}]})
```

Пользователь, запустивший операцию, и администраторы по умолчанию автоматически добавляются в результирующий ACL операции.

## Как правильно использовать ACL { #acl_usage}

Старайтесь задавать ACL как можно выше по дереву Кипариса и использовать наследование для потомков. Такой подход делает модель доступа предсказуемой и уменьшает число повторяющихся правил на отдельных таблицах.

Рекомендуется соблюдать следующие правила:

1. Выдавайте права группам, а не отдельным пользователям.
2. Используйте `deny` только как исключение, например для срочного ограничения уже открытого доступа. После этого пересмотрите структуру ACL и по возможности вернитесь к модели, основанной на `allow`.
3. Отключайте наследование ACL только там, где модель доступа действительно меняется.
4. Проверяйте критичные изменения с помощью `check-permission` до применения на боевых путях.

Для чувствительных поддеревьев, например каталогов с токенами или другой конфиденциальной метаинформацией, задавайте правила доступа явно и не полагайтесь на настройки, унаследованные от верхних уровней.
