# Debugging API calls in HTTP and RPC proxies

This article describes a practical setup for debugging API calls in {{product-name}}:
- request-by-request structured logs on proxies;
- trace propagation and forced sampling;
- OpenTelemetry usage notes.

## 1. Enable structured request logs for all proxy calls

For request-level debugging, use structured logs:
- HTTP proxy: `HttpStructuredProxy` category (`structured_http_proxy_log` in docs).
- RPC proxy: `RpcProxyStructuredMain` category (`structured_rpc_proxy_log` in docs).

{% note warning %}

RPC proxy main structured logs can be heavy because they may include request bodies. Use rotation and size limits.

{% endnote %}

### HTTP proxy

Configure a structured logger for category `HttpStructuredProxy` in the HTTP proxy logging config.

### RPC proxy

Configure a structured logger for category `RpcProxyStructuredMain` in the RPC proxy logging config.

If your goal is literally “all calls”, make sure the dynamic config does not suppress methods:

```bash
yt set //sys/rpc_proxies/@config '{
  api = {
    structured_logging_main_topic = {
      enable = %true;
      suppressed_methods = [];
      methods = {};
    };
  };
}'
```

By default, RPC proxy may suppress high-frequency methods in main topic logs.

## 2. Correlate logs between clients and proxies

Useful IDs in proxy logs:
- `request_id` — per-request ID generated by proxy/client stack.
- `correlation_id` — stable client-side ID across retries (HTTP proxy log).
- `trace_id` — distributed tracing ID (present when tracing context exists).

For HTTP API calls, pass `X-YT-Correlation-ID` header from the client to simplify retry correlation.

## 3. Force tracing while debugging

During incidents, it is often useful to force sampling temporarily.

### HTTP proxy

```bash
yt set //sys/http_proxies/@config '{api = {force_tracing = %true;}}'
```

### RPC proxy

```bash
yt set //sys/rpc_proxies/@config '{api = {force_tracing = %true;}}'
```

You can also tune proxy tracing sampler settings in dynamic config under `tracing`.

## 4. Is OpenTelemetry tracing possible?

Short answer: **yes on client side, not as a native proxy tracing backend**.

- Proxy/server tracing in {{product-name}} is built around Jaeger/OpenTracing configuration.
- Go SDK supports OpenTelemetry integration via OpenTracing bridge (`ytotel.TraceFn`), see `yt/go/examples/tracing/main.go`.

So the common approach is:
1. keep proxy tracing/logging in built-in format (Jaeger/OpenTracing + structured logs);
2. use OpenTelemetry in clients and bridge context into YT requests.

## 5. Recommended incident workflow

1. Enable/verify structured logs on HTTP and RPC proxies.
2. Reproduce the failing call with a fixed correlation ID.
3. Find entries by `correlation_id`, `request_id`, or `trace_id`.
4. For RPC proxy, temporarily unsuppress methods if needed.
5. Temporarily enable `force_tracing` and inspect distributed trace chain.
6. Disable aggressive debugging settings after incident resolution.
